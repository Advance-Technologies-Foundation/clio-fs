# Clio FS Operator UI ‚Äî Design System

This document defines the visual language, design tokens, component vocabulary, and implementation conventions for the operator UI. Every UI contribution must follow this guide to maintain consistency with the Creatio product family that this tool supports.

> **Reference:** Review the live Creatio product at `https://bundle.krylov.cloud` (login: `Supervisor` / `Supervisor1!`) to calibrate visual expectations before contributing.
>
> - Grid/list: https://bundle.krylov.cloud/Shell/#Section/Contacts_ListPage
> - Card detail: https://bundle.krylov.cloud/Shell/#Card/Contacts_FormPage/edit/c4ed336c-3e9b-40fe-8b82-5632476472b4
> - Tab styles and badges: https://bundle.krylov.cloud/Shell/#Card/Opportunities_FormPage/edit/6344d6ce-f889-4361-83f2-9cd71dbd6300

## Implementation model

This is **not a React app**. The UI is fully **server-rendered HTML** generated by TypeScript template-literal functions. There is no client-side framework, no build step for the UI, and no CSS files on disk. All HTML, CSS, and inline JavaScript are emitted by two TypeScript modules:

| Module | Role |
|--------|------|
| `packages/ui-kit/src/index.ts` | CSS tokens + all reusable render functions (the "component library") |
| `apps/server-ui/src/server.ts` | Page-level assembly: routes, data fetching, composing render functions into full pages |

The UI is served by a plain Node.js HTTP server (`node:http`). Any change to visual output requires editing those two files, rebuilding the package, and restarting the server.

## 1. Brand Alignment

The dashboard is an operator tool for Creatio instances. Users are Creatio developers and platform engineers who work daily inside the Creatio product. The dashboard must feel like a natural extension of that product ‚Äî not a foreign DevOps tool bolted on.

**Key visual traits from the Creatio product:**
- Top navigation bar: very dark, almost-black surface with the Creatio logo on the left and user controls on the right
- Content area: clean white or very light gray, high readability
- Sidebar navigation: dark purple-black surface with colored section icons
- Primary action buttons ("Save"): solid blue, no gradient, medium border-radius
- Secondary/cancel buttons: white background, border, no fill ‚Äî the button label speaks for itself
- Status chips/badges: small rounded pills with colored backgrounds; Customer = orange, Employee = teal
- Form tab navigation: underline indicator, not pill or card tabs
- Typography: clean, modern sans-serif ‚Äî we use **Montserrat** throughout
- Spacing: generous whitespace; the product never feels cramped

## 2. Design Tokens

Tokens live in the `:root` block inside `renderPage()` in `packages/ui-kit/src/index.ts`. There are no separate CSS files. Every page response carries the full token set embedded in a `<style>` tag in the document `<head>`.

To change a token value, edit the `:root` block in `renderPage()`.

## 4. Component Inventory

All UI components are TypeScript functions that return HTML strings. They live in `packages/ui-kit/src/index.ts`. There are no `.tsx` files, no React imports, and no JSX anywhere in this project.

### 4.1 Buttons

Three CSS classes, applied directly to `<button>` elements in template literals:

| Class | Use case | Background | Height |
|-------|----------|------------|--------|
| `.primary-button` | Create, Apply (form submit) | `--color-primary` blue | `40px` |
| `.secondary-button` | Cancel, Choose Folder | white + border | `36px` |
| `.danger-button` | Delete | `--color-danger` red | `36px` |

All buttons use `--radius-md` (8 px). Disabled state: `opacity: 0.5; cursor: not-allowed`.

```html
<button type="submit" class="primary-button">Create workspace</button>
<button type="button" class="secondary-button">Cancel</button>
<button type="button" class="danger-button">Delete</button>
```

**Rule:** Cancel always appears to the **left** of the primary/danger action in `.modal-actions`.

---

### 4.2 Status Badge ‚Äî `renderStatusBadge(status)`

**File:** `packages/ui-kit/src/index.ts`  
**Signature:** `renderStatusBadge(status: WorkspaceStatus): string`

Returns an inline-styled `<span>` pill. Colors are driven by `badgeStyles` map at the top of the file.

| Status | Background | Text |
|--------|------------|------|
| `active` | `--color-success-soft` green | `--color-success-text` |
| `disabled` | `--color-warning-soft` amber | `--color-warning-text` |
| *(fallback)* | gray | dark gray |

Shape: `border-radius: 9999px`, padding `2px 10px`, `font-size: 0.75rem`, bold, uppercase.

To add a new `WorkspaceStatus` value, add an entry to `badgeStyles` in `packages/ui-kit/src/index.ts`.

---

### 4.3 Notice Banner ‚Äî `renderNotice(tone, message)`

**Signature:** `renderNotice(tone: "error" | "success", message: string): string`

Returns a `.notice-banner` div. Used inline on the dashboard when a form POST produces an error or success message.

| Tone | Class | Color |
|------|-------|-------|
| `error` | `.notice-error` | `--color-danger-soft` background, `--color-danger-text` text |
| `success` | `.notice-success` | `--color-success-soft` background, `--color-success-text` text |

```ts
renderNotice("error", "Workspace ID already exists")
```

---

### 4.4 Metric Card ‚Äî `renderMetricCard(label, value)`

**Signature:** `renderMetricCard(label: string, value: string): string`

A `.panel` tile with a `.metric` label and `.metric-value` number/string. Used in the `.grid` stat strip at the top of each page.

```ts
renderMetricCard("Workspaces", String(workspaces.length))
renderMetricCard("Health", health.status)
```

---

### 4.5 Workspace Table ‚Äî `renderWorkspaceTable(items)`

**Signature:** `renderWorkspaceTable(items: WorkspaceRecord[]): string`

Renders a `.table-card` with a standard `<table>`. Columns: Name (link to detail), Status (badge), Revision, Delete button.  
When `items` is empty, renders a `.table-card` with a `.empty` message instead.

Also renders the `<dialog data-delete-dialog>` delete-confirmation modal that is wired up by the inline script in `renderPage()`.

---

### 4.6 Workspace Registration Form ‚Äî `renderWorkspaceRegistrationForm(values?, platform?)`

**Signature:** `renderWorkspaceRegistrationForm(values?, serverPlatform?): string`

A `.panel` containing a `.form-grid` form with four fields:

| Field | Input | Notes |
|-------|-------|-------|
| Workspace ID | text, required | auto-populated from folder name by inline JS |
| Display Name | text | optional |
| Root Path | text + Choose Folder button | populated by `/native/select-directory` |
| Platform | text, readonly | set server-side, not editable |

Submits `POST /workspaces/register`. Pass `values` to re-populate fields after a validation error.

---

### 4.7 Page Shell ‚Äî `renderPage(title, body)`

**Signature:** `renderPage(title: string, body: string): string`

The outermost wrapper. Returns a full `<!doctype html>` document that includes:
- Google Fonts (`Montserrat`)
- All CSS (tokens + component styles) in a `<style>` tag
- Fixed `.topbar` with orange dot, "Clio FS" title, "Control Plane" subtitle
- `.shell` content wrapper
- Inline `<script>` for folder picker and delete dialog behaviour

Every route handler calls `renderPage()` as its outermost wrapper.

---

### 4.8 Utilities ‚Äî `formatWorkspaceLabel` / `escapeHtml`

- `formatWorkspaceLabel` ‚Äî returns `"Display Name (workspaceId)"` when a display name is set, otherwise just `workspaceId`. Use in all workspace name displays.
- `escapeHtml` ‚Äî escapes `& < > " '`. **Always call this before interpolating user-supplied strings into HTML.**

---

## 5. Visual Patterns

### Link style

All hyperlinks use `--color-text-link`, underline on hover. Workspace name links in the table are the primary navigation action.

### Detail rows (DL)

Workspace detail uses a `.meta-list` `<dl>` layout:
- `dt`: `--font-size-xs`, uppercase, `--color-text-secondary`, monospaced letter-spacing
- `dd`: `--font-size-sm`, `--color-text-primary`, wraps gracefully

### Section kicker / eyebrow

Small uppercase label above a page heading. Use the `.eyebrow` class ‚Äî `--font-size-xs`, `--color-text-secondary`, `--font-weight-bold`, letter-spacing `0.12em`.

```html
<div class="eyebrow">Workspace Detail</div>
<h1>my-workspace</h1>
```

### Inline `<code>` fragments

Use bare `<code>` tags. CSS gives them `--color-primary-soft` background, `--color-primary` text, and `--radius-sm` corners. Use for workspace IDs and path values inline in prose.

---

## 6. Writing Style

- **Eyebrow kickers:** ALL CAPS, concise ‚Äî `WORKSPACE DETAIL`, `CLIO FS OPERATOR SURFACE`
- **Headings (`h1`):** Short and direct ‚Äî use workspace ID or label as-is
- **Metric labels:** ALL CAPS, very short ‚Äî `SERVICE`, `HEALTH`, `PLATFORM`, `WORKSPACES`
- **Button labels:** Sentence case ‚Äî `Create workspace`, `Choose Folder`, `Cancel`, `Delete`
- **Empty states:** lowercase, present-tense ‚Äî `no workspaces registered yet.`
- **Badge text:** ALL CAPS ‚Äî `ACTIVE`, `DISABLED`

## 7. Do / Don't

| Do | Don't |
|----|-------|
| Use `.primary-button` (blue) for form submit | Hardcode `background: #1463C8` outside a CSS class |
| Use `Montserrat` via Google Fonts link in `renderPage()` | Use system fonts (`Segoe UI`, Georgia) |
| Use flat white `.panel` card surfaces | Use `backdrop-filter: blur` anywhere |
| Keep `.modal-actions`: Cancel left, danger/primary right | Swap Cancel and action button positions |
| Call `escapeHtml()` on every user string before output | Interpolate raw user values directly into HTML |
| Add new badge styles to `badgeStyles` map in `ui-kit` | Inline badge colors in `server.ts` |
| Keep all CSS inside `renderPage()` in `ui-kit` | Add `<style>` blocks in `server.ts` page fragments |

## 9. CSS Conventions

- All CSS lives in the `<style>` block inside `renderPage()` in `packages/ui-kit/src/index.ts`. There is no external CSS file.
- Use CSS custom properties (`var(--color-*)`, `var(--radius-*)`, etc.) ‚Äî never hardcode color hex values.
- CSS sections inside `renderPage()` are separated by `/* --- Section --- */` banner comments. Maintain this order: tokens ‚Üí reset ‚Üí topbar ‚Üí shell ‚Üí hero ‚Üí stat grid ‚Üí panel ‚Üí metric ‚Üí nav ‚Üí table-card ‚Üí empty ‚Üí meta-list ‚Üí form ‚Üí buttons ‚Üí notice ‚Üí modal ‚Üí responsive.
- Breakpoints: `720px` (stack to single column).
- Inline `style=""` attributes are acceptable for one-off layout overrides (e.g., `style="margin-bottom:0;"`), but colors must still reference CSS variables.
- Never introduce CSS modules, Sass, PostCSS, or any CSS preprocessor.

## 10. HTTP Routes

All routes are handled in `apps/server-ui/src/server.ts` inside `createServerUi()`. The server talks to the control-plane API at `controlPlaneBaseUrl` using a Bearer token.

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/` | Dashboard: health + stat cards + registration form + workspace table |
| `GET` | `/workspaces/:id` | Workspace detail: revision, status, policies meta-list |
| `POST` | `/workspaces/register` | Register a workspace; redirects to detail on success, re-renders form on error |
| `POST` | `/workspaces/:id/delete` | Delete a workspace; redirects to `/` |
| `POST` | `/native/select-directory` | Opens a native OS folder picker; returns `{ path }` JSON or `204` on cancel |

The control-plane URL and auth token come from `@clio-fs/config` (`appConfig.serverUi`). In dev, defaults are `http://127.0.0.1:4010` with token `dev-token`.

## 11. Development Workflow

```powershell
# From the repo root ‚Äî builds all packages then starts the control-plane + UI:
corepack pnpm -r build
node scripts/dev-server.mjs
```

Or use the root script shorthand:

```powershell
corepack pnpm run server
```

Once running:

| Service | URL |
|---------|-----|
| Operator UI | http://127.0.0.1:4020 |
| Control Plane API | http://127.0.0.1:4010 |

After editing any `.ts` file in `packages/ui-kit` or `apps/server-ui`, rebuild and restart:

```powershell
corepack pnpm -r build && node scripts/dev-server.mjs
```

There is no hot-reload. The dev loop is: edit ‚Üí build ‚Üí restart ‚Üí refresh browser.

## 12. Deployment

The UI is served by the Node.js `@clio-fs/server-ui` process. To deploy:

1. Run `corepack pnpm -r build` to compile all TypeScript to `dist/`.
2. Start the process: `node apps/server-ui/dist/index.js`.

Configure the control-plane URL and auth token via the `@clio-fs/config` package values (or override them in a future environment-variable layer). No static file serving, no CDN, no separate frontend build step is required.
